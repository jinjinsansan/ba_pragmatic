# AI Vision カードデータ取得 設計メモ

最終更新: 2026-05-10

---

## 背景・仮説

7000ゲーム以上の運用経験から以下の仮説あり：

- **バンカー優勢シュー**: バンカーの最初の2枚が5以上になることが多く、8/9ナチュラルで勝つことが多い
- **プレイヤー優勢シュー**: プレイヤーの最初の2枚が5以上になることが多く、8/9ナチュラルで勝つことが多い

→「初期2枚の合計がどちら側に偏るか」がシューの性格を決めているという仮説。  
→ これを数値で検証・AI学習の特徴量として活用したい。

---

## なぜ WS では取れないか

Pragmatic の game WS は以下しか送信しない：
- `betsopen` / `betsclosed`（BET窓開閉）
- `timer`（タイマー）
- `bet`（BET確認）

実際の勝敗はロビー DGA フィード（`winner` のみ）から取得。  
カードデータはゲーム画面（ブラウザ iframe 内部）にしか存在せず、WSには乗ってこない。  
（2026-05-10 の `[card-debug]` ログ調査で確認済み）

## 採用アプローチ: Claude Vision API

### フロー
```
ラウンド終了（winner 確定後）
→ Playwright でカード表示部分をスクリーンショット（クロップ）
→ Claude Haiku Vision API に送信
→ レスポンスから p_card1, p_card2, b_card1, b_card2 を抽出
→ DB に保存
→ マスターUI にリアルタイム反映（1〜3秒後）
```

### なぜ Vision か
- クロスオリジン制限なし（JS インジェクションは iframe がサードパーティドメインのため50%以下の成功率）
- 追加ライブラリ不要（Playwright で screenshot 可能、Anthropic API は既存）
- バカラのカードは読みやすくHaikuで十分な精度が期待できる

### API コスト試算
| モデル | 1ゲーム | 1日500ゲーム | 月額 |
|---|---|---|---|
| Claude claude-haiku-4-5（推奨） | ~$0.0002 | ~$0.10 | ~$3 |
| Claude Sonnet（高精度） | ~$0.0015 | ~$0.75 | ~$22 |

→ まず Haiku でテスト。精度が足りなければ Sonnet に切替。

## 取得するデータ（1ラウンドごと）

| カラム名 | 内容 |
|---|---|
| `p_card1` | プレイヤー1枚目（0〜9） |
| `p_card2` | プレイヤー2枚目（0〜9） |
| `p_initial` | プレイヤー初期合計（0〜9） |
| `b_card1` | バンカー1枚目（0〜9） |
| `b_card2` | バンカー2枚目（0〜9） |
| `b_initial` | バンカー初期合計（0〜9） |
| `p_natural` | プレイヤーナチュラル（初期合計8/9） |
| `b_natural` | バンカーナチュラル（初期合計8/9） |
| `p_card3` | プレイヤー3枚目（引いた場合のみ） |
| `b_card3` | バンカー3枚目（引いた場合のみ） |

## シュー単位の集計（AI学習特徴量）

```
banker_initial_avg   ← シュー内バンカー初期合計の平均
player_initial_avg   ← シュー内プレイヤー初期合計の平均
banker_natural_rate  ← バンカーナチュラル率
player_natural_rate  ← プレイヤーナチュラル率
shoe_banker_wr       ← そのシューのバンカー勝率

→ banker_initial_avg と shoe_banker_wr の相関を検証
```

---

## 注意事項

- スクリーンショットはラウンド結果確定後（`winner` 受信後）に撮る（カードが全て表示された状態）
- API 呼び出しは非同期で行い、BET 判断をブロックしないこと
- Vision 失敗時は `null` を記録してスキップ（BET ロジックには影響させない）
- 3枚目カードはオプション（引かない場合もあるため NULL 許容）
